# DOSDisk — заметки проекта

## Что это
Один `index.html`: js-dos v8 (DOSBox в WASM) + загрузка диска из GitHub-репо/ссылки.

## Состав (упрощено по запросу)


- **GitHub как диск**: `list()` = git trees recursive; `read()` = git/blobs (base64); `write()` =
  contents API (PUT, base64, sha существующего файла). `list`+`read` → store-zip bundle → boot.
- **Загрузка с компьютера** (drag-drop / выбор): файл пишется И в живой эмулятор (`ci.fsWriteFile`),
  И в репозиторий (`GitHub.write`).
- **Синхронизация**: `dumpFs()` (обход `ci.fsTree`/`fsReadFile`) → `GitHub.write` по каждому файлу.
- **Полный экран**: Fullscreen API на `#dos` (кнопка ⛶), повторное нажатие — выход.

## Все библиотеки — локально (без чужих серверов)
js-dos лежит в `vendor/js-dos/` (скачан из npm `js-dos@8.3.20`):
- `js-dos.js`, `js-dos.css` — движок и стили;
- `emulators/` — рантайм эмулятора: `emulators.js` + WASM (`wdosbox.wasm` 1.4 МБ,
  `wdosbox-x.wasm` 6.6 МБ, `wlibzip.wasm`) и их `.js`-загрузчики.
js-dos ищет WASM по опции **`pathPrefix`** — в `boot()` задаём `pathPrefix:'vendor/js-dos/emulators/'`.
По умолчанию он смотрел на `https://v8.js-dos.com/...`, который отдаёт 404 (отсюда была ошибка
`Dos is not defined` — сам js-dos.js тоже грузился с мёртвого CDN).
Единственные внешние обращения клиента — `api.github.com` (это сам сервис, не библиотека).

## Мультидиск (GitDOS): как заработало (важно!)
Ограничения движка js-dos в этой сборке, выясненные опытным путём:
1. **Только один смонтированный диск C:** — второй `mount` валит DOSBox. Поэтому доп.
   репозитории кладём папками внутрь C:: `C:\DISKS\<буква>` (`buildDrives`: folder=`DISKS/<L>`).
2. **`zip_to_fs` (распаковка bundle) НЕ создаёт подпапки** — bundle с `D/file` падает
   («unable to write file D/...» → ExitStatus). Поэтому в bundle кладём только файлы КОРНЯ C:.
3. **Подпапки дописываем после старта** через `ci.fsWriteFile` (он умеет createPath/mkdir).
   Но DOSBox кэширует список диска при монтировании и не видит дописанное; `RESCAN` (только с Z:)
   кэш чистит, но драйв всё равно не подхватывает — тупик.
4. **РЕШЕНИЕ:** после дозаписи делаем `ci.persist(false)` (родной формат js-dos, с подпапками)
   и `boot()` из него — C: **перемонтируется заново**, уже со всеми папками. Видно всё. ✓
   (см. `loadPending` в index.html). Синхронизация — по папке диска (`relOf` в `pushChanges`).

## Ключевые решения
- **Движок js-dos v8, а не классический dosbox.js.** Классический (`Dosbox`+`js-dos-v3.js`) умеет
  только смонтировать zip и запустить — нет API для выгрузки изменённых файлов. v8 даёт
  `fsTree()`/`fsReadFile()` — без них запись назад невозможна.
- **GitHub как диск:** читаем дерево (`git/trees/{branch}?recursive=1`), тянем содержимое блобов
  (`git/blobs/{sha}`, base64) → собираем js-dos **bundle** (обычный zip) с `.jsdos/dosbox.conf`,
  где autoexec монтирует `c .` и (опц.) запускает указанный exe → грузим как `url` в `Dos()`.
- **Свой store-zip** (CRC32 + метод 0, без сжатия) — js-dos сам распаковывает, сжатие не нужно.
- **Выгрузка (фаза 2):** обходим `ci.fsTree()`, читаем `ci.fsReadFile(path)` → ZIP на скачивание.

## Безопасность (важно)
- GitHub-токен вводится в браузере, уходит только на api.github.com. Не коммитим, не логируем.


## Грабли / на проверку
- API js-dos v8 (`onEvent` события `ci-ready`/`emu-ready`, форма `fsTree`) проверить руками —
  при расхождении поправить по факту отчёта стажёра.
- Прямые ссылки на чужие zip упираются в CORS. GitHub raw/API — без проблем.
- Большой репозиторий = много блобов = долгая загрузка (по одному запросу на файл).
  Если станет узким местом — перейти на скачивание tarball архива репо одним запросом.
- На `file://` ненадёжно — запускать через http-сервер / Pages.
